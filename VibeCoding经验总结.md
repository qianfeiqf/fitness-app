# Vibe Coding 经验总结：用 Claude Code 重构微信小程序

> 基于一个健身小程序的完整 UI/UX 重构实践，涵盖视觉重设计、组件开发、库迁移、代码清理等全链路。

---

## 一、项目背景

**项目**：微信小程序健身训练计划记录工具
**起点**：已有完整功能代码，使用 TDesign 组件库，"单色+红色强调"设计风格
**目标**：纯黑白极简 + 新拟物化 (Neumorphism) 视觉重设计，同时清理技术债务
**规模**：约 35 个文件修改 + 8 个新建文件，跨 12 个页面 + 5 个自定义组件

---

## 二、工作流程与节奏

### 2.1 分阶段推进

整个项目自然分成了几个阶段，每个阶段有明确的交付物：

```
Phase 0: 地基 — Token 变量、全局样式、图标替换
Phase 1: 新组件 — dial-picker、modal-sheet 等自定义组件
Phase 2A: 简单页面 — 主要是颜色替换（stats、history、complete）
Phase 2B: 中等页面 — 阴影 + 颜色（plans、exercises、profile、onboarding）
Phase 2C: 复杂页面 — 大幅重构（home、training）
Phase 3: 技术迁移 — TDesign → WeUI + 原生 API
Phase 4: 代码清理 — 内联弹窗组件化、死代码删除
```

**经验**：让 Claude 自己制定分阶段计划（用 `/plan` 命令），然后逐阶段执行。每完成一个阶段就验证，不要一口气改完所有文件。

### 2.2 一次做一类事情

最高效的模式是：**明确一个目标 → Claude 执行 → 验证 → 下一个目标**。

好的指令示例：
- "把 training 页面的 stepper 输入替换为 dial-picker 组件"
- "把所有 JSON 文件中的 TDesign 注册清除"
- "把 home 页的内联弹窗替换为 modal-sheet 组件"

不好的指令示例：
- "把整个 app 重做"（太模糊，Claude 会不知道从哪开始）
- "优化一下代码"（没有具体目标）

## 三、有效的 Prompt 模式

### 3.1 给足上下文，但聚焦目标

```
✅ "把 training.wxml 里的 <t-toast>、<t-dialog> 删掉，
    因为 tdesign.js 已经改用原生 wx.* API 了，
    这些组件不再需要渲染在页面上"

❌ "删掉不需要的东西"
```

Claude 不知道你的意图是什么。"不需要的东西"对人来说很明显，对 AI 来说需要明确说明**为什么不需要**。

### 3.2 用截图反馈迭代

这次项目中最有效的迭代模式：

1. Claude 改了代码
2. 我在微信开发者工具中预览，截图
3. 把截图发给 Claude，指出问题
4. Claude 修复

**dial-picker 指示线位置**经历了 4 轮这样的迭代：
- 第1轮：线在文字中间 → "横线应该在文字底部"
- 第2轮：距离太大 → 调小 offset
- 第3轮：启用 snap 后线又穿过了文字 → 调整计算逻辑
- 第4轮：最终确认 OK

**经验**：截图是最高效的反馈方式。一句"看图，线还是穿过了文字中间"比描述 CSS 属性值更准确。

### 3.3 引导 Claude 做审计，而不是直接改

迁移 TDesign 时，先让 Claude 审计：
- "哪些文件还在用 TDesign？"
- "哪些 TDesign 组件注册其实没有被使用？"
- "哪些手写代码可以用 WeUI 替换？"

Claude 输出审计报告后，再让它按优先级执行。这比直接说"把 TDesign 换掉"更可控。

### 3.4 利用 Task 系统追踪进度

```
用户: "按优先级高低进行替换"
Claude: 创建 7 个 Task，逐个执行并标记完成
```

这让复杂的多步骤工作变得可追踪。尤其当上下文被压缩（context compaction）时，Task 列表是唯一保留的进度信息。

## 四、踩过的坑

### 4.1 路径问题

组件注册时写了 `"../../../components/modal-sheet/modal-sheet"`，但 `pages/home/home` 只有两层深，应该是 `../../components/...`。

**教训**：让 Claude 改完 JSON 后，立即在开发者工具中编译验证，不要等全部改完再试。

### 4.2 scroll-view 的 scroll-top 与 scroll-snap 冲突

dial-picker 组件中，`scroll-snap-type: y mandatory` 会覆盖程序化的 `scroll-top` 设置。解决方案是：
- 用户手势时：CSS snap 负责吸附
- 程序化跳转时：用 `scroll-top` 定位

两者共存的关键是：`scroll-snap` 只在用户手势滚动时生效，`scroll-top` 的程序化设置不受 snap 影响。

**教训**：微信小程序的 scroll-view 行为和 Web 不完全一致。让 Claude 先调研再写代码，而不是先写再调。

### 4.3 同值 scroll-top 不触发重渲染

设置 `scroll-top` 为同一个值不会触发视图更新。解决方法是先设为 0，再在 `wx.nextTick` 中设为目标值。

### 4.4 原生 button 有默认样式

微信的 `<button>` 元素默认是绿色背景 + 圆角。用 `.btn` class 覆盖时，还需要清除 `::after` 伪元素的边框：

```css
button.btn::after { border: none; }
```

### 4.5 Claude 会"忘记"之前的上下文

当对话很长时，上下文会被压缩。这时 Claude 可能会重复已经做过的事情，或者忘记之前的决策。

**应对方法**：
- 用 Task 系统记录已完成的工作
- 关键决策让 Claude 写入 `CLAUDE.md` 或 memory 文件
- 每次新对话开始时，简要说明当前状态

## 五、最高效的协作模式

### 5.1 "你来规划，我来审批"

```
用户: /plan 把 TDesign 换成 WeUI
Claude: [输出详细计划，列出每个文件的改动]
用户: OK，执行
Claude: [逐个文件执行]
```

这比用户逐个文件指挥高效得多。Claude 能看到整个代码库，规划出最优的执行顺序。

### 5.2 "先审计，再动手"

```
用户: 查一下哪些写死的代码可以用微信组件替换
Claude: [输出审计报告，按优先级分类]
用户: 按优先级高低进行替换
```

让 Claude 先理解现状，再给出方案，最后执行。跳过审计直接动手，往往会遗漏或做错。

### 5.3 "截图驱动开发"

```
用户: [截图] 还有问题，线还是穿过文字中间了
Claude: 调整 calc(50% + 20rpx) → calc(50% + 10rpx)
用户: [截图] 这个距离似乎太大了
Claude: 再调 → calc(50% + 15rpx)
```

视觉类的工作，截图反馈是不可替代的。

### 5.4 善用"子任务"

```
用户: 再帮我开一个子任务查看一下哪些写死的代码也可以用微信组件替换
```

当主线任务进行中，发现需要额外调研时，开一个子任务（Agent）并行处理。主任务不中断。

## 六、经验清单（可复用的 Checklist）

### 开始新项目/新功能前

- [ ] 让 Claude 先用 `/plan` 输出执行计划
- [ ] 确认计划后再开始编码
- [ ] 如有设计稿/截图，一开始就提供

### 视觉/UI 类工作

- [ ] 每改一个页面，立即在模拟器中预览
- [ ] 用截图反馈，而不是用文字描述 CSS 问题
- [ ] 全局 Token（颜色、阴影、字体）先改，再改页面
- [ ] 组件先做好，再在页面中引用

### 迁移/重构类工作

- [ ] 先审计：哪些文件用了旧方案？哪些注册是死代码？
- [ ] 先改配置/注册，再改代码
- [ ] 改完后立即编译验证，不要攒到最后
- [ ] 清理残留：CSS 变量、JSON 注册、npm 依赖

### 组件开发

- [ ] 先确认 API 设计（属性、事件），再写实现
- [ ] 路径用相对于页面的路径，注意目录层级
- [ ] 微信小程序的 scroll-view、button 等原生组件有默认样式，需要 reset

### 调试技巧

- [ ] scroll-top 不生效 → 检查是否与 scroll-snap 冲突
- [ ] 同值 setData 不刷新 → 先清零再设值
- [ ] 原生 button 有绿色背景 → 清除 ::after 伪元素
- [ ] 组件路径报错 → 检查相对路径层级是否正确

## 八、一句话总结

> **Vibe Coding 的核心不是"让 AI 写代码"，而是"让 AI 理解你的意图，然后你验证它的输出"。截图、分阶段、先审计再动手——这三个习惯让效率翻倍。**
